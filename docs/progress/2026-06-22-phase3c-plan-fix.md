# 2026-06-22 — Phase 3c: 플랜 한도 수정 + 저장 구조 개선

## 문제
- 사용자는 **Pro** 요금제인데 기본값이 Max5였음.
- Pro로 바꾸니 5시간이 100%(다 씀)로 표시 — 실제론 39%인데.
- 원인 1: Pro 프리셋 한도(0.85M)가 실제(약 4.0M)보다 5배 낮은 엉터리 추정.
- 원인 2: 한도가 data.json에 **저장**돼서 프리셋 코드를 고쳐도 반영 안 됨.

## 진단 (diagnose.ts)
현재 측정(가중): 5시간 1.572M, 주간 2.140M.
- Claude 실측 39% → 5시간 한도 = 1.572 / 0.39 ≈ 4.0M
- Claude 실측 16% → 주간 한도 = 2.140 / 0.16 ≈ 13.4M

## 해결 1: 저장 구조 변경 (저장값 → 계산값)
- 이전: settings에 fiveHourLimit/weeklyLimit를 저장 → 프리셋 수정이 묻힘.
- 변경: **plan만 저장**, 한도는 `effectiveLimits()`가 PLAN_LIMITS에서 계산.
  custom일 때만 customFiveHourLimit/customWeeklyLimit 사용.
- 효과: 프리셋 코드를 고치면 즉시 반영. 기존 data.json(plan=pro)도 새 Pro 한도 자동 적용.

## 해결 2: 플랜 프리셋 실측 보정
PLAN_LIMITS (가중 토큰):
- pro:   5h 4.0M  / 주간 13.4M   (실측 anchor)
- max5:  5h 20M   / 주간 67M     (Pro x5)
- max20: 5h 80M   / 주간 268M    (Pro x20)

## 배운 것
- **추정 한도는 정말 추정일 뿐** — 내 초기 플랜 배수는 완전히 틀렸고, 실측 한 점(39%)이
  훨씬 정확했음. 사용자의 ground truth가 최고의 보정 수단.
- **파생 가능한 값은 저장하지 말 것**: 한도는 plan에서 계산 가능 → 저장하면 코드 수정이
  사용자 데이터에 묻혀버리는 stale 문제 발생. plan만 저장하는 게 옳음.
- 디버깅 교훈: "예전에 Max5로 우연히 맞았던" 것은 내 Max5 추정치가 진짜 Pro 한도와
  비슷했던 우연. 우연한 일치를 신호로 착각하면 안 됨.
