# 2026-06-22 — Phase 3d: 비용가중 → 비캐시 토큰 metric

## 계기
사용자 질문: Claude 통계 화면은 7일 총 토큰이 ~400K인데, 우리는 5시간 한도를 4M으로
잡고 있다. 4M이 말이 되냐?

## 분석 (단위 혼동이었음)
- "400K" = Claude 통계 화면의 **비캐시 토큰** (cache_read 제외)
- "4M" = 우리 **비용가중 단위** (cache_read를 ×0.1로 깎아 포함). 둘은 다른 단위 → 비교 불가.
- 4M은 "측정값이 39%가 되게 만드는 분모"일 뿐, 진짜 토큰 한도가 아님.

데이터 점 2개로 한도 역산:
| 시점 | Claude% | 비캐시 | 가중 |
|------|---------|--------|------|
| 이전 | 32% | 0.237M | 1.238M |
| 최근 | 39% | 0.284M | 1.572M |
- 가중 한도: 3.87M / 4.03M (일관)
- 비캐시 한도: 0.74M / 0.73M (더 일관)

→ 두 metric 다 게이지를 잘 따라감. 그러나 **비캐시가 더 직관적**(Claude 화면과 같은 단위)
이고 절대값이 믿을 만함 → 비캐시로 전환.

## 변경
- `usage.ts`: `entryTokens = input + output + cacheCreate` (cache_read 제외, 가중 제거)
- `settings.ts` PLAN_LIMITS(비캐시 단위): pro 5h 0.74M/주간 3.5M, max5 ×5, max20 ×20
- 레이블 "가중" → "비캐시"
- README 동작 방식 갱신

## 배운 것
- **단위를 맞춰라**: 외부 숫자(Claude 통계 406K)와 내 숫자를 비교하려면 같은 정의여야 함.
  비교 불가능한 단위를 섞으면 "4M이 이상하다" 같은 혼란 발생.
- **데이터 점이 여러 개면 metric 검증 가능**: 한 점은 어떤 metric이든 맞출 수 있지만,
  두 점이 같은 한도를 가리키면 그 metric이 일관적이라는 증거.
- 단순하고 해석 가능한 metric(비캐시 합)이, 복잡하지만 불투명한 가중합보다 나을 수 있음.

## 한계
- 비캐시도 추정. Anthropic 실제 한도가 cache_read를 일부 친다면 과소집계 가능.
- 여전히 한 시점 보정이라 cache 비율 변동에 ±몇% 흔들림.
