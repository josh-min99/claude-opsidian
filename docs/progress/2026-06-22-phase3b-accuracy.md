# 2026-06-22 — Phase 3b: % 정확도 수정 (비용 가중 + 보정)

## 문제
- Claude 앱은 5시간 32%인데 플러그인은 52%로 과대 표시. 주간도 2~3% 높게 잡힘.

## 원인 (진단으로 확인)
현재 5시간 블록 토큰 구성:
- input 0.002M / output 0.082M / cacheCreate 0.153M / **cacheRead 6.349M (96%)**

→ 토큰을 단순 합산하니 **cache 읽기가 96%를 지배**. 그런데 Claude 실제 한도는
cache 읽기를 크게 할인(비용 ~0.1x)함. 그래서 우리 수치만 뻥튀기됨.
주간이 그나마 맞아 보인 건 여러 세션이 섞여 cache 비율이 평균화된 우연.

## 해결: 비용 가중 metric
`usage.ts`의 `entryTokens`에 가중치 적용:
- input ×1, output ×5, cacheCreate ×1.25, cacheRead ×0.1 (Anthropic 과금 비율 근사)

적용 후: 5시간 6.59M → **1.33M**, 주간도 동반 감소.

## 해결: 사용자 보정 기능
한도는 여전히 추정치라, 설정에 **"Claude 실제 % 입력 → 한도 역산"** 추가.
- 현재 측정값(가중) 표시 + % 입력 → `한도 = 측정값 / (% / 100)`
- 입력 시 plan이 자동으로 "직접 입력"으로 전환됨
- 예: 5시간 1.33M, Claude 32% 입력 → 한도 4.16M 자동 설정

## 배운 것
- **토큰은 다 같은 토큰이 아니다**: cache read/write, input, output의 비용·한도 가중치가 다름.
- 외부 시스템(Claude)의 숫자와 맞출 땐, 내가 가진 ground truth(실제 %)로 **역산 보정**하는 게
  추측보다 정확하고 사용자 친화적.
- 디버깅: 두 지표 중 하나만 틀리면(주간 OK, 5h 틀림) "분자(계산)"보다 "분모(한도)"나
  "구성 비율"을 의심 → 구성 분해(diagnose.ts)가 결정적이었음.
